# O 號程式查找機制（簡化假設）

對應 `docs/PLAN.md` 第13節第13點。

## 目前的簡化假設

`gcode_sim.program_registry.ProgramRegistry` 把整份輸入原始碼依
`O<number>` 開頭的行切成多個區塊，每個區塊各自 parse 成一份獨立的
statement 列表，登記在同一個 registry 裡：

- 沒有 O 號的內容（例如檔案最前面、第一個 O 行之前的部分）如果只是
  空白/換行，會被忽略，不會產生一個空的「主程式」蓋掉真正的程式——這是
  Phase 1 開發過程中發現的真實 bug（見下方「開發筆記」）。
- 若整份原始碼完全沒有任何 `O` 開頭的行，整份內容視為唯一的「主程式」。
- 若有多個 O 號區塊，`main_program()` 回傳**第一個**被登記的區塊
  （不論它有沒有 O 號）。也就是說，主程式應該放在檔案最前面。
- `M98 Pnnnn` 呼叫時，依 `nnnn` 數值到 registry 裡查找對應的 O 號程式。

## 明確不實作的部分

- 真機的「依資料夾搜尋順序尋找同名程式」機制（手冊有專門章節，但不在
  我們讀到的兩份手冊範圍內）。
- 多檔案合併：目前 `ProgramRegistry.register_source()` 只接受單一字串
  來源。若之後需要「主程式一個檔案、宏程式庫另外放」的情境（業界常見
  做法），`cli.py` 可以擴充 `--macro-lib dir/` 之類的參數，把多個檔案的
  內容個別呼叫 `register_source()` 累加進同一個 registry——這不需要改
  `ProgramRegistry` 本身的資料結構，只需要在呼叫端多呼叫幾次。
- 同名 O 號重複定義時的行為：目前後定義的會覆蓋先定義的（Python dict
  的自然行為），沒有報錯或警告。

## 開發筆記：Phase 1 曾經踩到的坑

初版 `_split_by_o_number` 實作裡，只要「目前累積的行數不是空」就會被
當成一個區塊押進 chunks，包括 O 號行之前純粹是空白/換行的部分。這代表
像下面這種很常見的寫法（Python 三引號字串開頭就會多一個空行）：
```
O0001 ;
#1=0 ;
...
```
會被誤判成「有一個沒有 O 號的空白主程式」+「一個 O0001 的程式」，
而 `main_program()` 優先回傳沒有 O 號的那個（因為它被視為「主程式」
的預設身分），結果拿到的是空程式，M98/GOTO 等完全沒有執行到任何東西，
但也不會報錯——是那種「安靜地什麼都沒做」的錯誤，很難從報錯訊息看出
問題在哪。修法是在 `_flush()` 裡先檢查 `body.strip()` 是否為空，空白
內容不算數，避免產生這種「假的主程式」。這也是 `tests/
test_control_flow.py` 裡好幾個測試最一開始跑起來直接輸出「EMPTY」而
不是報錯的原因——本來看起來像是變量沒被正確賦值，實際上是整支程式
根本沒被執行到。
